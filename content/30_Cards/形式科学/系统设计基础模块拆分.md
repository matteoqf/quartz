# 系统设计基础模块拆分

模块拆分是系统设计中的核心实践，旨在将复杂系统分解为可管理、可维护、可扩展的单元。

## 模块拆分原则

### 1. 单一职责原则

每个模块只负责一项明确的业务功能，避免职责混杂。

**判断标准**：
- 模块能否用一句话描述其职责
- 是否存在多个变化的原因
- 修改该模块是否会影响其他不相关功能

### 2. 高内聚低耦合

**高内聚**：模块内部元素联系紧密，相关功能集中放置
**低耦合**：模块间依赖关系最小化，仅通过稳定接口交互

**耦合类型（从低到高）**：
| 类型 | 特征 |
|------|------|
| 非直接耦合 | 无直接关系，通过消息通信 |
| 数据耦合 | 通过参数传递数据 |
| 标记耦合 | 传递数据结构 |
| 控制耦合 | 传递控制信息 |
| 外部耦合 | 依赖外部环境 |
| 公共耦合 | 共享全局数据 |
| 内容耦合 | 直接访问内部数据（最差） |

### 3. 层次分明

系统按层级组织，每层只与上下相邻层交互：

```
┌─────────────────────────────┐
│       表示层 (UI)           │
├─────────────────────────────┤
│       应用层 (Service)       │
├─────────────────────────────┤
│       领域层 (Domain)        │
├─────────────────────────────┤
│       基础设施层 (Infra)     │
└─────────────────────────────┘
```

### 4. 领域驱动设计

根据业务领域划分模块边界，而非技术实现：

**示例（电商系统）**：
- 客户模块：客户注册、认证、信息管理
- 商品模块：商品发布、上下架、库存管理
- 订单模块：下单、支付、履约、售后
- 营销模块：优惠券、活动、积分

## 常见拆分模式

### 1. 按业务能力拆分

根据业务职能划分模块，适用于业务边界清晰的系统。

### 2. 按领域模型拆分

以核心领域概念为模块单位，强调领域知识的内聚。

### 3. 按技术分层拆分

按基础设施、应用逻辑、接口协议等垂直分层。

### 4. 按部署单元拆分

根据扩展性和部署需求划分独立可部署的模块。

## 模块接口设计

### 稳定性原则

模块接口应比内部实现更稳定：

- 暴露接口，隐藏实现细节
- 接口变更应向后兼容
- 优先使用数据传递，避免共享状态

### 粒度控制

- **过粗**：模块过大，职责不清晰，难以复用
- **过细**：模块过多，调用链复杂，集成成本高

**经验法则**：模块能被单独理解、测试、部署为宜。

## 拆分的演进思维

### 初期阶段
- 单体架构快速验证
- 模块化设计，但部署不分离

### 增长阶段
- 按业务优先级逐步拆分
- 识别高频变更与稳定模块

### 成熟阶段
- 微服务或模块化单体
- 独立部署、独立扩展

## 常见问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 循环依赖 | 模块划分不清 | 引入中介层或重构边界 |
| 过度设计 | 预测过多变化 | YAGNI原则，先简化 |
| 分布式谬误 | 假设模块会部署在不同机器 | 按需拆分，避免为拆分而拆分 |

## 总结

模块拆分没有绝对标准，关键在于：

1. **职责清晰**：每个模块有明确边界
2. **依赖可控**：通过稳定接口交互
3. **易于测试**：模块可独立验证
4. **便于演进**：支持渐进式扩展

拆分是持续优化的过程，应随业务复杂度迭代调整。
