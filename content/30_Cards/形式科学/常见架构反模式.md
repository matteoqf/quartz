# 常见架构反模式

## 概述

架构反模式是指在软件架构设计中常见的、会导致系统难以维护、扩展或演进的错误模式。这些模式通常在短期内看似合理，但长期会带来技术债务和系统退化。

---

## 一、逻辑架构反模式

### 1. 分布式单体 (Distributed Monolith)

**问题描述**：将单体应用拆分为多个服务，但服务之间高度耦合，任何变更都需要多个服务同时部署。

**特征**：
- 服务间同步调用链过长
- 一个服务的失败导致级联失败
- 部署需要协调多个服务
- 共享数据库或共享 schema

**解决方向**：识别真正的业务边界，采用事件驱动解耦，或在必要时合并回真正的小型单体。

---

### 2. 瑞士军刀 (Swiss Army Knife)

**问题描述**：创建一个过于通用、承担过多职责的组件，试图满足所有可能的用例。

**特征**：
- 类或模块承担数十种不相关的功能
- 接口参数列表过长且复杂
- 代码难以理解和测试
- 高度耦合，牵一发动全身

**解决方向**：遵循单一职责原则，拆分通用组件为多个专注的小组件。

---

### 3. 玻璃瓶 (God Object/Class)

**问题描述**：某个类或模块集中了过多的数据和行为，成为系统的中心瓶颈。

**特征**：
- 单个类文件超过数千行
- 大量实例变量和实例方法
- 其他类直接依赖该类
- 修改该类风险极高

**解决方向**：识别领域概念，通过职责剥离和领域建模分散职责。

---

## 二、组件架构反模式

### 4. 循环依赖 (Circular Dependency)

**问题描述**：组件 A 依赖 B，B 依赖 C，C 又依赖 A，形成依赖环。

**特征**：
- 模块间 import 关系形成环
- 无法独立编译和测试任何单个组件
- 构建顺序难以确定
- 修改一个模块影响所有相关模块

**解决方向**：通过依赖倒置、接口隔离或依赖注入打破循环，重构为单向依赖图。

---

### 5. 依赖chaos (Dependency Chaos)

**问题描述**：组件之间的依赖关系无明确规则和层次，混乱交织。

**特征**：
- 跨层调用普遍存在
- 高层组件依赖低层组件
- 依赖深度无限制
- 无法确定代码的所有权边界

**解决方向**：建立清晰的层次结构规则，如整洁架构的依赖规则，采用依赖检查工具强制执行。

---

### 6. 过深继承树 (Shotgun Surgery / Feature Envy)

**问题描述**：继承层次过深或过宽，导致理解困难和脆弱的变更。

**特征**：
- 继承层次超过 3-4 层
- 子类覆盖大量父类方法
- 新功能需要理解整个继承链
- 修改父类影响不可预测

**解决方向**：优先使用组合而非继承，采用策略模式或模板方法重构过深继承。

---

## 三、分布式系统反模式

### 7. 盲目微服务 (Microservices without Need)

**问题描述**：将系统过早或过度拆分为微服务，增加了不必要的复杂性。

**特征**：
- 每个微服务只有 1-2 个功能
- 服务间通信开销超过业务逻辑本身
- 分布式事务成为常态
- 运维复杂度指数级增长

**解决方向**：先构建模块化单体，在证明有明确需要时才拆分为服务。优先考虑领域驱动设计划分边界。

---

### 8. 共享数据库 (Shared Database)

**问题描述**：多个服务或组件直接共享同一个数据库，各自随意读写。

**特征**：
- 服务间通过数据库间接耦合
- 一个服务的查询影响其他服务性能
- 无法独立演进数据模型
- 字段含义在多个服务中重复定义

**解决方向**：为每个服务提供独立的数据存储，通过 API 或事件进行交互。

---

### 9. 超时和重试chaos

**问题描述**：缺乏统一的超时和重试策略，导致系统行为不可预测。

**特征**：
- 不同服务使用不同的超时值
- 重试没有限制和退避策略
- 级联超时导致系统假死
- 无熔断机制

**解决方向**：建立超时和重试策略标准库，实施熔断和限流模式。

---

## 四、数据架构反模式

### 10. 读写一体 (Read/Write Coupling)

**问题描述**：同一个数据模型同时承担读写职责，导致读写互相影响。

**特征**：
- 复杂的 ORM 映射关系
- 读写操作互相阻塞
- 难以单独优化读或写路径
- 缓存策略难以实施

**解决方向**：采用 CQRS 模式分离读写模型，使用事件溯源记录状态变更。

---

### 11. 无Schema演化

**问题描述**：数据库 schema 变更缺乏版本管理和兼容性考虑。

**特征**：
- 字段随意添加删除
- 无向后兼容策略
- 历史数据无法正确解释
- 数据库迁移缺乏测试

**解决方向**：采用渐进式 schema 变更策略，保持向后兼容，使用 expand-contract 模式。

---

### 12. 报告干扰事务 (Reporting干扰 Transaction)

**问题描述**：将报表查询和分析操作与核心业务事务混合，影响系统性能。

**特征**：
- 复杂查询拖慢核心事务
- 报表与业务数据紧耦合
- 无法独立扩展报表能力
- 索引策略难以同时优化两类需求

**解决方向**：采用读写分离或独立报表数据库，通过事件同步业务数据。

---

## 五、集成架构反模式

### 13. 硬编码外部依赖

**问题描述**：系统中硬编码了对外部服务、数据库或配置的引用。

**特征**：
- 配置文件散布各处
- 外部 URL 直接写在代码中
- 无法在不同环境间切换
- 测试需要真实外部系统

**解决方向**：实施配置中心化，统一管理所有外部依赖引用，使用依赖注入提高可测试性。

---

### 14. API 版本chaos

**问题描述**：多个 API 版本同时存在，缺乏版本管理策略。

**特征**：
- 客户端使用各种版本的 API
- 无法废弃旧版本
- 兼容逻辑复杂且脆弱
- 文档与实现不一致

**解决方向**：制定明确的 API 版本策略，采用语义化版本管理，设置合理的废弃过渡期。

---

### 15. 缺乏边缘计算逻辑

**问题描述**：所有业务逻辑集中在中心节点，忽视边缘计算能力。

**特征**：
- 每次交互都需要往返中心服务器
- 网络延迟影响用户体验
- 中心节点负载过重
- 无法离线工作

**解决方向**：评估将适当逻辑下沉到边缘的可行性，考虑 CDN 边缘函数或客户端智能。

---

## 六、识别和应对

### 识别方法

1. **代码度量**：圈复杂度、继承深度、耦合度
2. **架构适应度函数**：自动化测试架构规则合规性
3. **设计评审**：定期架构评审会议
4. **技术债务指数**：跟踪技术债务累积趋势

### 应对策略

- **渐进式重构**：不要试图一次性解决所有问题
- **影响评估**：量化每个反模式的风险和修复成本
- **持续监控**：建立架构健康度指标体系
- **团队共识**：确保团队对架构原则有共同理解
