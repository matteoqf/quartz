# 数据库索引数据结构原理

type: principle

## 核心问题

在磁盘上存储的大量数据中快速定位目标记录，核心矛盾是**磁盘IO次数与数据量大小的关系**。顺序扫描的时间复杂度为 O(n)，而索引将查找降低到 O(log n) 甚至 O(1)。

---

## 索引的本质

索引是一种**额外的数据结构**，存储「查找键 → 数据位置」的映射关系，使得无需逐行扫描全表即可定位目标。

---

## 常见索引数据结构

### 1. B-Tree 索引

**结构**：多路平衡查找树，每个节点包含多个键值对，子节点按范围划分。

```
        [50, 70]
       /   |   \
  [<50]  [50-70]  [>70]
```

**特点**：
- 所有叶子节点深度相同，查找时间复杂度稳定 O(log_m n)
- 节点大小通常等于磁盘页大小（4KB/8KB/16KB），最大化IO效率
- 支持范围查询和顺序访问
- 插入/删除后通过分裂/合并保持平衡

**变种**：
- B+Tree：非叶子节点仅存储键，叶子节点存储全部数据且用链表串联，利于范围扫描
- B*Tree：节点至少2/3满，减少分裂次数

**数据库支持**：MySQL(InnoDB)、PostgreSQL、Oracle — 默认索引类型

---

### 2. 哈希索引

**结构**：基于哈希表，键通过哈希函数映射到桶（bucket），每个桶内存储数据指针。

**特点**：
- 等值查询时间复杂度 O(1)
- **不支持范围查询、排序**
- 哈希冲突导致链表/红黑树解决

**变种**：
- 动态哈希表：桶可动态扩展，避免性能退化
- 局部性哈希：对热点数据建立小型哈希索引

**数据库支持**：Memory引擎、Redis

---

### 3. 位图索引（Bitmap Index）

**结构**：每个键值对应一个位图，位图中每一位对应一行，1表示该行该键值存在。

```
键值 "北京" → 101010010...
键值 "上海" → 010101001...
```

**特点**：
- 空间紧凑，适合低基数（枚举值少）字段
- 支持快速交、并、补位运算
- 更新代价高，锁竞争严重

**数据库支持**：Oracle、SQL Server

---

### 4. 倒排索引（Inverted Index）

**结构**：词项 → 文档列表的映射，支持快速全文检索。

```
"数据库" → [doc1, doc3, doc7]
"索引"   → [doc1, doc2, doc5]
```

**特点**：
- 支持分词、TF-IDF、BM25等文本相关性计算
- 压缩存储：增量编码、FXP编码等

**数据库支持**：Elasticsearch、Solr

---

### 5. R-Tree 索引

**结构**：多维空间数据的层次索引，每个节点用最小边界矩形（MBR）描述子节点覆盖范围。

**特点**：
- 支持多维范围查询、最近邻查询
- 常用于地理空间数据

**数据库支持**：PostgreSQL(PostGIS)、MySQL(Spatial)

---

## 磁盘IO与索引设计

### 关键洞察

- 磁盘IO是顺序读取优于随机IO
- 磁盘页（16KB典型）是一次读写单位
- B-Tree的节点大小对齐磁盘页，使每次IO收益最大化

### 索引层次计算

给定：
- 磁盘页大小 = 16KB
- 索引键大小 = 8B
- 指针大小 = 8B
- 每个节点可存储键数 ≈ 16KB / 16B ≈ 1000

则：
- 3层 B+Tree 可索引：1000 × 1000 × 1000 = 10亿条记录
- 查找只需3次磁盘IO

## 索引的代价

| 代价类型 | 说明 |
|---------|------|
| 空间成本 | 索引结构占用额外存储，通常为表数据的10%-30% |
| 写操作开销 | 每次INSERT/UPDATE/DELETE需同步更新索引 |
| 维护成本 | 数据变更导致节点分裂/合并，积累碎片 |

---

## 索引设计原则

1. **选择性（Selectivity）**：区分度高的列优先建立索引，低基数列（如性别、状态）不适合B-Tree
2. **查询模式匹配**：索引应覆盖WHERE、JOIN、ORDER BY中出现的列
3. **前缀索引**：长字符串列可使用前缀匹配减少空间
4. **复合索引最左前缀**：索引 (A,B,C) 可加速 A=、A+B=、A+B+C= 的查询
