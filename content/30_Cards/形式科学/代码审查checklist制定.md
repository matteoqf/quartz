# 代码审查 Checklist 制定

## 基本概念

代码审查（Code Review）是通过系统化地检查源代码，发现缺陷、改进质量并促进知识共享的协作实践。一份结构良好的审查清单（Checklist）是确保审查一致性和覆盖面的工具。

**前置条件：** 待审查代码已提交至代码仓库，审查者已获取审查范围说明。

**后续结果：** 审查意见记录于协作工具中，作者完成修改并通过复核。

## 流程步骤

### 1. 确定审查目标与范围

审查启动时，首先明确本次审查的目标：

- **目标定义**：修复 bug、新功能实现、重构、代码清理、安全审计等
- **范围界定**：明确审查涉及的模块、文件与变更量
- **角色确认**：作者（提交者）、审查者（一人或多人）、主持人（如需会议审查）

> 变更量过大的 PR（超过 400 行）建议拆分为多次审查。

### 2. 准备审查清单

根据变更类型，从标准清单库中选取适用的检查项，并视情况补充本次审查的特殊关注点。

#### 通用检查项（所有变更必查）

| # | 检查项 | 检查要点 |
|---|--------|----------|
| 1 | 代码格式规范 | 缩进、空格、换行符合项目编码风格；无残留调试代码 |
| 2 | 命名一致性 | 变量/函数/类命名符合项目约定；无拼写错误；同一概念用同一名称 |
| 3 | 注释与文档 | 公共 API 有文档注释；复杂逻辑有解释；无过期或误导性注释 |
| 4 | 错误处理 | 错误被捕获并妥善处理；错误信息有意义；关键路径有降级方案 |
| 5 | 日志与监控 | 关键操作有日志记录；日志级别恰当；不含敏感信息 |
| 6 | 测试覆盖 | 新增代码有对应单元测试；测试用例合理；不引入回归 |
| 7 | 依赖变更 | 依赖版本升级有理由说明；无引入不必要的新依赖 |
| 8 | 配置与常量 | 硬编码值抽取为配置或常量；敏感信息不硬编码在代码中 |
| 9 | 提交信息规范 | commit message 清晰描述变更内容与动机 |
| 10 | 安全基线 | 无硬编码密钥/凭证；输入经验证；无已知漏洞依赖 |

#### 按变更类型选查

**功能变更：**
- 业务逻辑正确性：实现是否符合需求规格
- 边界条件处理：空值、极值、并发场景
- 向后兼容性：是否破坏已有接口或数据格式

**性能相关：**
- 循环与算法复杂度：是否存在隐性 O(n²) 或更高
- 数据库查询：是否有效利用索引；是否有 N+1 查询问题
- 资源释放：连接、文件句柄、内存是否及时释放

**安全相关：**
- 输入验证与消毒
- 权限校验（认证与授权）
- 敏感数据处理与加密
- SQL/命令注入防护

**重构相关：**
- 行为保持：重构后功能未改变
- 无重复代码
- 模块边界清晰，依赖方向正确

### 3. 执行审查

按清单逐项检查，遵循以下原则：

- **先逻辑，后细节**：先理解代码意图和架构，再检查语法和风格
- **客观描述**：用事实而非感受描述问题，引用代码而非泛泛而谈
- **建设性优先**：指出问题的同时给出改进建议
- **分类优先级**：将问题标记为 Blocking（阻止合并）、Recommended（建议修改）或 Optional（可选优化）

### 4. 记录与跟进

- 将审查意见记录至代码托管平台的 Review 系统（如 GitHub PR Review、GitLab Merge Request）
- 明确每个问题的处理预期（由作者修改、或接受并记录原因）
- 复核作者修改后的代码，确认问题已解决

### 5. 复盘与清单迭代

审查结束后，定期回顾审查记录：

- 是否有某类问题反复出现但清单未覆盖
- 清单中是否有检查项长期无问题，可降优先级
- 是否需要新增检查项应对新引入的技术或规范

## 清单管理建议

- 将标准清单维护在团队共享文档或仓库的 `.github/CODE_REVIEW_CHECKLIST.md` 中
- 根据项目阶段和技术栈裁剪清单（初创项目侧重功能正确性，成熟项目侧重安全与性能）
- 审查者和作者共同遵守清单，确保审查质量可预期

## 常见反模式

- **走过场式审查**：机械勾选清单，不深入分析逻辑
- **清单过于冗长**：检查项超过 20 项时审查者疲劳，建议分层（必查/建议查/按需查）
- **只关注风格问题**：格式问题可由自动化工具（linter）处理，审查应聚焦于工具无法发现的问题
- **缺乏上下文**：仅贴代码行号而不解释问题影响，导致沟通低效

## 流程示意

```
确定目标与范围 → 准备清单 → 执行审查 → 记录意见 → 作者修改 → 复核 → 清单迭代
```
