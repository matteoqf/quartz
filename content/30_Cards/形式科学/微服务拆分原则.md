# 微服务拆分原则

微服务拆分原则是指导单体应用向微服务架构演进的核心准则，旨在实现服务的高内聚、低耦合，以及系统的可维护性、可扩展性和技术灵活性。

## 单一职责原则 (SRP)

**定义**：每个微服务只负责一个明确的业务功能域，拥有独立的数据和业务逻辑。

**核心思想**：服务边界对应业务边界，避免服务职责重叠或模糊。

**优点**：
- 服务独立部署和扩展
- 故障隔离，减少级联影响
- 团队自治，按服务组建开发团队
- 技术异构，允许不同服务使用不同技术栈

**示例**：
```
# 合理的服务划分
├── 用户服务 (User Service)
│   └── 负责：用户注册、登录、权限管理
├── 订单服务 (Order Service)
│   └── 负责：订单创建、查询、取消
└── 支付服务 (Payment Service)
    └── 负责：支付流程、退款、账务
```

## 领域驱动设计原则 (DDD)

**定义**：基于业务领域概念和模型进行服务拆分，以业务边界而非技术分层作为划分依据。

**核心思想**：将业务复杂度与技术复杂度分离，通过领域模型捕捉业务本质。

**关键概念**：
- **聚合 (Aggregate)**：一组紧密相关的实体和值对象，作为数据修改的单元
- **限界上下文 (Bounded Context)**：业务领域的边界，每个上下文拥有独立的领域模型
- **防腐层 (Anti-Corruption Layer)**：在不同服务/系统间转换数据模型

**示例**：
```
电商领域划分
├── 商品上下文 (Product Context)
│   └── 商品、库存、分类
├── 交易上下文 (Trading Context)
│   └── 订单、促销、优惠券
└── 履约上下文 (Fulfillment Context)
    └── 物流、仓储、配送
```

## 低耦合高内聚原则

**定义**：服务间依赖最小化，服务内部元素关联最大化。

**核心思想**：
- **低耦合**：服务间通过标准接口通信，不共享数据库，实现独立演化
- **高内聚**：相关功能放在同一服务内，减少跨服务调用

**实现方式**：
- 避免服务间直接共享数据表
- 使用事件驱动替代同步调用
- 采用Saga模式处理跨服务事务

## 独立部署原则

**定义**：每个微服务可独立构建、测试和部署，不依赖其他服务的部署顺序或版本。

**核心思想**：服务之间通过API契约通信，而非代码级依赖。

**优点**：
- 支持持续集成和持续部署 (CI/CD)
- 支持灰度发布和蓝绿部署
- 故障恢复快，可单独回滚问题服务

## 前后置分离原则

**定义**：将核心业务逻辑与周边功能（日志、监控、安全）分离。

**核心思想**：
- 核心服务专注业务逻辑
- 横切关注点通过 Sidecar/Proxy 统一处理

**实现方式**：
- 服务网格 (Service Mesh) 处理可观测性、安全和流量管理
- API Gateway 统一处理认证、限流、协议转换

## 粒度适中原则

**定义**：服务粒度应平衡团队规模、业务复杂度和技术可行性。

**核心思想**：
- 服务过粗：导致服务复杂、团队耦合、部署冲突
- 服务过细：导致服务膨胀、分布式系统复杂度激增、事务难以管理

**经验法则**：
- 2-Pizza Team 原则：一个服务团队 6-10 人能完整维护
- 服务数量参考：初期 5-15 个，逐步演进
- 数据库数量与服务数量相当

## 演进式拆分原则

**定义**：微服务拆分是渐进过程，而非一次性重构。

**核心思想**：
- 优先拆分稳定、独立、高变化频率的模块
- 识别团队技能和业务优先级
- 保持过度状态，允许单体与微服务共存

**实施步骤**：
1. 识别业务领域和依赖关系
2. 定义服务边界和API契约
3. 逐步迁移，从边缘服务开始
4. 监控和调优，持续改进
